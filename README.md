# Нейросимулятор Смоленска — проблема Б

Локальный аналитический прототип моделирует транспортную доступность и безопасность городской мобильности. Единственный расчётный источник истины — `colab/colab_kirill/Pipeline.ipynb`; FastAPI переносит его подготовку данных, восемь нечётких индексов, итоговую линейную свёртку и ANFIS. FCM используется поверх Pipeline как отдельный сценарный модуль.

Архитектурный план и статус внедрения: [`Document/ПЛАН_УЛУЧШЕНИЙ.md`](Document/ПЛАН_УЛУЧШЕНИЙ.md).
Актуальная расчётная архитектура Pipeline: [`Document/PIPELINE_АРХИТЕКТУРА.md`](Document/PIPELINE_АРХИТЕКТУРА.md).

## Данные и расчётный pipeline

В `datasets_ready` поставляются два совместимых источника, лист `Лист1`:

- `smolensk_dataset_shared.xlsx` — основной набор из 80 кварталов;
- `smolensk_dataset_plus_zero.xlsx` — его демонстрационная копия с кварталом `2026Q1`, где все 31 входных показателя равны нулю.

Раздел «Данные для расчёта» позволяет выбрать активный XLSX, просмотреть любую строку, изменить все 31 показателя или добавить следующий квартал. Период и пять служебных полей создаются автоматически, а файл заменяется только после полной проверки Pipeline. Наблюдатель имеет доступ на чтение; пользователь и администратор могут выбирать и редактировать источник.

Переключение в интерфейсе действует до перезапуска приложения. Стартовый файл задаётся переменной `SMOLENSK_DATA_PATH` в `.env`; по умолчанию используется основной набор.

- 80 исходных кварталов: 2006Q1–2025Q4;
- 5 служебных полей и 31 числовой показатель;
- двухуровневый заголовок с шестью тематическими блоками;
- после IQR-проверки исключается выброс средней скорости за `2018Q3`, поэтому в расчётах остаётся 79 строк.

Pipeline:

```text
Excel → проверка схемы и порядка 31 показателя
      → 6 преобразований log1p + удаление выброса 2018Q3
      → 8 нечётких индексов Pipeline
      → линейная свёртка с экспертными весами → target
      → RobustScaler → Seasonal Naive / Ridge / ANFIS
      → FCM-сценарии безопасности / регулярности / доступности
```

Разбиение по времени выполняется без перемешивания:

- train: 2006Q1–2018Q4, 51 строка после исключения `2018Q3`;
- validation: 2019Q1–2022Q4, 16 строк;
- test: 2023Q1–2025Q4, 12 строк.

`log1p` применяется к дворам, двум пассажиропотокам, ДТП блока городского транспорта, переходам того же блока и завершённым мероприятиям. Коробки с усами и IQR-границы доступны в разделе 8 интерфейса.

## Colab-методы

Из `Pipeline.ipynb` перенесены `LinguisticVariable`, треугольные и трапециевидные функции принадлежности, Mamdani-агрегация и centroid-дефаззификация на сетке из 1000 точек. Шесть канонических JSON-файлов применяются в восьми системах и дают 1 176 активных правил суммарно. Диапазоны функций заданы в исходных единицах notebook, а не в общей искусственной шкале.

Рассчитываются индексы современной городской среды, доступной среды, общественных пространств, парковок и безопасности, а также качество и благополучие дорог для двух транспортных блоков.

Восемь нечётких индексов напрямую агрегируются в `target` с весами `0.6 / 0.6 / 0.7 / 0.2 / 0.5 / 0.7 / 0.9 / 0.4`, нормированными на сумму `4.6`. Повторная min-max-нормализация индексов перед этой свёрткой не выполняется — это соответствует классу `LinearConvolutionIndex` в Pipeline.

ANFIS перенесён без обязательных PyTorch, scikit-learn и joblib: NumPy-реализация использует восемь входов, шесть гауссовых правил Сугено, RobustScaler, Adam, раннюю остановку и параметры `epochs=500`, `lr=0.05`, `patience=70`. Для стабильной воспроизводимости линейные консеквенты инициализируются формулой target, поскольку target по определению является линейной свёрткой тех же восьми входов.

Исходные `anfis_best.pt` и `*.pkl` сервером не исполняются: pickle-артефакты небезопасно загружать из непроверенного источника. Внутренние компактные ANFIS-калибраторы FCM по-прежнему кэшируются как проверяемые `.npz`, но в разделе сравнения моделей показывается только модель Pipeline.

Канонический `colab/colab_kirill/Pipeline.ipynb` и правила `colab/colab_kirill/*_rules.json` версионируются в Git. В Docker-образ копируются необходимые runtime JSON-правила; сам notebook для запуска сервера не требуется. Backend валидирует имена всех входов, термов и выходов и завершает запуск с понятной ошибкой, если файл отсутствует или повреждён. Контрольные значения восьми индексов и target совпадают с сохранёнными outputs `Pipeline.ipynb`.

## Цель и модели

В проверке моделей используется одна цель `pipeline_target` — взвешенная свёртка восьми нечётких индексов. Безопасность, регулярность и доступность остаются выходами сценарного FCM-модуля и не подменяют цель notebook.

В разделе «Проверка моделей» каждый метод подписан по роли, входам и назначению:

| Метод | Роль | Как считает и что использует | Зачем нужен |
|---|---|---|---|
| Seasonal Naive — сезонный повтор | простой сезонный baseline | прогноз равен факту того же квартала прошлого года; используется только значение цели `t−4` | минимальная контрольная точка для сложных моделей |
| Ridge Regression | линейный статистический baseline | лаги `t−1`, `t−2`, `t−4` и среднее предыдущих четырёх кварталов | показывает возможности простой регрессии временного ряда |
| ANFIS | основная модель Pipeline | восемь нечётких индексов, шесть гауссовых правил и линейные консеквенты | воспроизводит модель из основного notebook |

Baseline — Seasonal Naive и Ridge; ANFIS является основной моделью. Метрики: MAE, RMSE, sMAPE, MASE и directional accuracy. FCM сравнивается отдельно в режимах `expert` и `adapted` только внутри сценарного анализа.

FCM содержит 16 узлов и 34 направленные связи. Обновление использует `alpha=0.35`, `lambda=1.3`; адаптированные веса рассчитываются как `0.70 × expert + 0.30 × data` с сохранением экспертного знака.

## Сценарии JSON

Встроены 25 сценариев: 10 точечных улучшений, 3 прямых и 2 обратных перераспределения, 8 сценариев причинных связей, инерционный и пользовательский варианты.

Пользовательский сценарий:

```json
{
  "version": 1,
  "id": "limited-budget-custom",
  "label": "Ограниченный бюджет",
  "description": "Пользовательская проверка",
  "mode": "adapted",
  "horizon": 8,
  "impulses": {
    "road_budget_execution": -0.15,
    "road_repair": -0.10
  }
}
```

Значения основных и дополнительных импульсов ограничены диапазоном `[-1, 1]`. Пользовательские сценарии сохраняются в PostgreSQL: основные поля — в отдельных колонках, `impulses` — в `JSONB`. Каждый сценарий имеет владельца; встроенные сценарии доступны всем ролям и не могут быть перезаписаны.

Сначала идут 10 точечных улучшений: один положительный управляемый фактор получает `+1`, остальные остаются нулевыми. Три прямых перераспределения дают выбранному бюджету `+1`, а у остальных управляемых факторов суммарно забирают `−1`. Два обратных варианта дают одному фактору `−1`, а остальным суммарно `+1`. Дополнительно заданы восемь понятных сценариев связей: ремонт дорог, состояние дорог, освещённость через прокси переходов, регулярность, цифровое управление через скорость и загруженность, время поездки, рост аварийности и баланс пешеходной инфраструктуры с дорожным пространством. В конце находятся инерционный и пользовательский варианты.

Пользователь или администратор может изменить импульсы, режим и горизонт прямо в интерфейсе и нажать «Сохранить». Для встроенного сценария создаётся пользовательская копия с новым идентификатором; для ранее сохранённого сценария обновляется существующая запись в PostgreSQL.

Основные API-маршруты после входа:

- `GET /api/metadata`, `/api/history`, `/api/indices`, `/api/analysis`, `/api/evaluation`;
- `GET /api/datasets`, `GET /api/datasets/{name}` — список XLSX и строки;
- `POST /api/datasets/select`, `POST/PUT /api/datasets/{name}/rows` — выбор источника, добавление и редактирование строк для пользователя и администратора;
- `GET /api/fcm?mode=adapted`;
- `GET/POST/PUT/DELETE /api/scenarios`;
- `GET/PUT /api/scenarios/{id}/shares` — выбор наблюдателей для пользовательского JSON;
- `POST /api/simulate`.

## Авторизация и роли

Используется локальный вход по логину и паролю без OIDC. Пароли хранятся как Argon2id-хеши, серверные сессии — в PostgreSQL, изменяющие запросы защищены CSRF-токеном.

- `observer` / Наблюдатель — получает упрощённый интерфейс без редакторов, но может раскрыть все девять аналитических разделов; видит сценарии, пять рекомендаций, проценты, бюджетный вывод, анализ Pipeline и оба XLSX только для чтения;
- `user` / Пользователь — создаёт свои JSON-сценарии, выбирает наблюдателей, переключает XLSX, редактирует строки и добавляет новый квартал;
- `admin` / Администратор — имеет те же бизнес-инструменты, видит владельцев сценариев, управляет пользователями и просматривает аудит изменений данных.

Публичной регистрации нет. Начальные демонстрационные аккаунты создаются командой `python -m app.manage seed-demo-users`, пароли берутся из `.env`.

## Бизнес-интерпретация

После запуска сценария интерфейс показывает относительно инерционного прогноза:

- изменение безопасности, регулярности, доступности и итоговой мобильности в процентах;
- изменение тех же индексов в пунктах и значения «было → стало»;
- расчётное снижение количества ДТП;
- ровно пять приоритетных предложений по выбранной цели, включая восемь индексов Pipeline и пример «Парковки и безопасность движения»;
- рейтинг трёх модельных бюджетных направлений по выбранной бизнес-цели.

Бюджетный рейтинг сравнивает одинаковое стандартизованное воздействие на исполнение дорожного, транспортного и безопасностного бюджета. Это приоритизация по поведению модели, а не финансовый ROI и не расчёт в рублях.

## Структура интерфейса

Аналитические разделы расположены в порядке, удобном для демонстрации заказчику:

1. **Лаборатория FCM-сценариев** — позволяет сравнить решения, увидеть эффект в процентах, сохранить JSON и адресно открыть его наблюдателям.
2. **Чувствительность** — ранжирует направления по ожидаемой силе влияния на выбранную цель.
3. **Текущее состояние** — даёт краткую сводку последнего доступного периода и исходную точку сравнения.
4. **История и сезонность** — помогает отличить устойчивую динамику от разового или сезонного изменения.
5. **Сводные индексы** — объединяет исходные показатели в понятные оценки от 0 до 100.
6. **Проверка моделей** — показывает качество прогноза на уже известных данных и границы неопределённости.
7. **Карта связей FCM** — наглядно объясняет связи между решениями, городскими факторами и целевыми показателями.
8. **Анализ данных Pipeline** — показывает коробки с усами, выбросы, треугольные и трапециевидные функции и веса итогового индекса.
9. **Данные для расчёта** — показывает активный XLSX, последние строки и редактор 31 показателя с добавлением следующего квартала.

Все разделы 1–9 при открытии страницы свёрнуты. Над ними расположен четырёхшаговый маршрут заказчика: выбрать цель, получить пять мер, проверить сценарий и уточнить данные. Под заголовком каждого находится заметная кнопка «Развернуть раздел» с поясняющим текстом; переход по ссылке из верхнего меню автоматически раскрывает нужный раздел. При раскрытии размеры графиков и FCM-карты пересчитываются под доступную ширину. Рядом с каждым заголовком есть подсказка «Что показывает?». Раздел 10 «Администрирование» показывается только администратору.

Для роли `observer` применяется отдельный упрощённый режим: лаборатория FCM стоит первой и раскрывается автоматически, чувствительность расположена второй. Наблюдатель может раскрыть любой из девяти разделов, выбрать встроенный или адресно открытый ему JSON, цель рекомендаций и XLSX для просмотра, увидеть владельца, запустить готовый расчёт и экспортировать файл. Изменение режима, горизонта, импульсов, XLSX и выдача доступа скрыты.

## Полная установка и запуск

### Что должно быть установлено

Для рекомендуемого запуска нужны:

- Windows 10/11 с WSL 2 и запущенным Docker Desktop;
- Git, если проект получается из репозитория;
- доступ к Docker Hub при первой сборке для загрузки базовых образов Python и PostgreSQL.

Python и виртуальное окружение для полного Docker-запуска не требуются: зависимости из `requirements.txt` устанавливаются внутрь образа приложения. `.venv` нужен только для локальной разработки по варианту 2.

Docker Desktop не устанавливается через `requirements.txt` и не входит в `.venv`. На Windows сначала установите и запустите [Docker Desktop](https://docs.docker.com/desktop/setup/install/windows-install/), затем проверьте:

```powershell
docker version
docker compose version
```

Если Docker Desktop завершает работу с сообщением `Access denied to Docker`, текущий пользователь не включён в локальную группу `docker-users`. Запустите PowerShell **от имени администратора**, выполните команду и затем выйдите из Windows и войдите снова:

```powershell
Add-LocalGroupMember -Group docker-users -Member $env:USERNAME
```

Членство в `docker-users` даёт возможность управлять контейнерами и должно предоставляться только доверенному локальному пользователю.

### Вариант 1. Весь проект в Docker Compose

Рекомендуемый способ для передачи и демонстрации. FastAPI и PostgreSQL запускаются в отдельных контейнерах, миграции и создание трёх аккаунтов выполняются автоматически.

Получите проект и перейдите в его каталог. Если репозиторий уже скачан, достаточно выполнить только `Set-Location`:

```powershell
git clone https://github.com/Cosmic-Man/neuro_simulator_smolensk.git
Set-Location neuro_simulator_smolensk
```

Создайте локальную конфигурацию, откройте её и замените пароль PostgreSQL и три демонстрационных пароля. Файл `.env` не попадёт в Git:

```powershell
Copy-Item .env.example .env
notepad .env
```

Запустите сборку и дождитесь состояния `healthy` у обоих сервисов:

```powershell
docker compose config --quiet
docker compose up --build -d
docker compose ps
```

Проверка API и последние сообщения приложения:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/health
docker compose logs --tail 100 app
```

Интерфейс: <http://127.0.0.1:8000>. Документация API: <http://127.0.0.1:8000/docs>.

Демонстрационные логины: `observer`, `user`, `admin`. Пароли заданы в локальном `.env`; перед передачей проекта их следует заменить.

Остановка без удаления данных:

```powershell
docker compose down
```

Первый запуск может занять несколько минут из-за загрузки и сборки образов. При старте обучается основная шестиправильная ANFIS Pipeline; три служебных калибратора FCM могут использовать проверяемый Docker-кэш `model_cache`.

### Вариант 2. FastAPI из `.venv`, PostgreSQL в Docker

Удобен для разработки и отладки Python-кода:

```powershell
Copy-Item .env.example .env
```


Если в `runtime/scenarios` остались JSON-файлы старой версии, после миграции их можно однократно перенести владельцу:

```powershell

```

### Что хранится в Git, а что остаётся только у клиента

В Git хранятся исходный код, миграции схемы, `compose.yaml`, `Dockerfile` и безопасный шаблон `.env.example`. В Git **не должны** попадать:

- рабочий `.env` с паролями;
- содержимое PostgreSQL;
- пользовательские JSON-сценарии, учётные записи, сессии и журнал аудита;
- локальный кэш обученных ANFIS-моделей;
- локальные каталоги `postgres_data`, `pgdata`, `runtime`, `backups`;
- дампы `*.dump`, `*.backup`, `*.sql`, `*.sql.gz`, файлы SQLite и локальные файлы `pgpass`.

PostgreSQL хранит данные в именованном Docker volume `neuro-simulator-smolensk_postgres_data`, а кэш ANFIS — в `neuro-simulator-smolensk_model_cache`; оба расположены вне каталога репозитория. Каталог `datasets_ready` подключается в контейнер как локальная папка, поэтому правки XLSX из интерфейса сохраняются после пересборки. Перед `git pull` сделайте копию изменённых клиентом XLSX: эти файлы являются рабочими данными, а не данными PostgreSQL. Не используйте `git add -f` для `.env`, дампов или каталогов данных.

В репозитории находятся все файлы развёртывания:

- `Dockerfile` и `.dockerignore` — образ FastAPI без локальных секретов и `.venv`;
- `compose.yaml` — отдельные сервисы приложения и PostgreSQL, health-check и постоянные volumes базы и моделей;
- `docker-entrypoint.sh` — автоматические миграции и начальные аккаунты;
- `.env.example` — безопасный шаблон локальных переменных;
- `alembic.ini` и `migrations/` — версионирование схемы PostgreSQL.

По умолчанию PostgreSQL доступен на `127.0.0.1:5432`. Если порт занят, измените `POSTGRES_PORT` в `.env`, например на `5433`.

### Обновление установленного проекта

Перед обновлением рекомендуется создать резервную копию. Затем получите изменения и пересоберите только изменившиеся слои. При старте приложение автоматически применит новые миграции PostgreSQL:

```powershell
git pull
docker compose up --build -d
docker compose ps
docker compose logs --tail 100 app
```

Обновление контейнера не меняет именованный volume PostgreSQL. Если менялись только значения `.env`, пересоздайте контейнеры без удаления данных:

```powershell
docker compose up -d --force-recreate
```

### Резервная копия PostgreSQL

Каталог `backups` исключён из Git. Следующие команды создают сжатый дамп внутри контейнера и копируют его на компьютер клиента без передачи в репозиторий:

```powershell
New-Item -ItemType Directory -Force .\backups | Out-Null
$backupName = "neuro_simulator_{0}.backup" -f (Get-Date -Format "yyyyMMdd_HHmmss")
docker compose exec postgres sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc -f /tmp/neuro_simulator.backup'
docker compose cp postgres:/tmp/neuro_simulator.backup ".\backups\$backupName"
docker compose exec postgres rm -f /tmp/neuro_simulator.backup
```

Храните резервные копии отдельно от каталога проекта. Восстановление дампа перезаписывает текущие данные и должно выполняться ответственным администратором после проверки выбранного файла.

### Полезные команды

```powershell
# Проверить состояние и посмотреть логи приложения
docker compose ps
docker compose logs -f app

# Пересобрать приложение после изменения исходников
docker compose up --build -d app

# Удалить контейнер, сохранив данные в Docker volume
docker compose down

# Удалить контейнер вместе с локальными данными PostgreSQL
# Внимание: эта команда необратимо удаляет базу демонстрационного проекта
docker compose down -v
```

`docker compose down -v` — единственная из приведённых штатных команд, которая удаляет именованный volume с клиентскими данными. Не используйте её при обновлении или обычной остановке.

Значения из `.env.example` предназначены только для локальной демонстрации. Для внешнего развёртывания пароли необходимо заменить, `SESSION_COOKIE_SECURE` включить при работе через HTTPS, а PostgreSQL не следует публиковать напрямую в интернет.
