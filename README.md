# Нейросимулятор Смоленска — проблема Б

Локальный аналитический прототип моделирует транспортную доступность и безопасность городской мобильности. Проект объединяет новый сводный датасет, две нечёткие свёртки из Colab, FCM, ANFIS и классические baseline-модели.

Архитектурный план и статус внедрения: [`Document/ПЛАН_УЛУЧШЕНИЙ.md`](Document/ПЛАН_УЛУЧШЕНИЙ.md).

## Данные и расчётный pipeline

Источник: `datasets_ready/smolensk_dataset_shared.xlsx`, лист `Лист1`.

- 80 кварталов: 2006Q1–2025Q4;
- 5 служебных полей и 31 числовой показатель;
- двухуровневый заголовок с шестью тематическими блоками;
- пустой исходный target заменяется воспроизводимыми расчётными целями.

Pipeline:

```text
Excel → проверка и train-only preprocessing
      → 8 нечётких индексов Кирилла
      → иерархический индекс Гульдар по 8 нечётким индексам
      → безопасность / регулярность / доступность
      → Seasonal Naive / Ridge / FCM / ANFIS
      → итоговый индекс и сценарный прогноз
```

Разбиение по времени выполняется без перемешивания:

- train: 2006Q1–2018Q4;
- validation: 2019Q1–2022Q4;
- test: 2023Q1–2025Q4.

Параметры нормализации, winsorization и моделей определяются без использования test.

## Colab-методы

Из notebook Кирилла перенесены `LinguisticVariable`, треугольные и трапециевидные функции принадлежности, Mamdani-агрегация, centroid-дефаззификация и все восемь таблиц правил — 2 164 правила суммарно.

Рассчитываются индексы современной городской среды, доступной среды, общественных пространств, парковок и безопасности, а также качество и благополучие дорог для двух транспортных блоков.

Из `idex_target.ipynb` перенесена экспертная свёртка: восемь нечётких индексов агрегируются в иерархический индекс с весами `0.6 / 0.6 / 0.7 / 0.2 / 0.5 / 0.7 / 0.9 / 0.4`. Входы нормализуются только по train. Иерархический индекс используется как независимый baseline и не заменяет три целевых ряда и согласованный итоговый индекс мобильности.

## Цели и модели

Три основные цели:

- безопасность движения — обратный нормализованный уровень ДТП;
- регулярность — средний показатель рейсов по расписанию;
- доступность — 30% регулярность + 20% скорость + 20% состояние дорог + 15% переходы + 15% нечёткая транспортная среда.



Сравниваются Seasonal Naive, Ridge по четырём лагам, экспертная FCM, адаптированная FCM и ANFIS. Метрики: MAE, RMSE, sMAPE, MASE и directional accuracy.

FCM содержит 16 узлов и 34 направленные связи. Обновление использует `alpha=0.35`, `lambda=1.3`; адаптированные веса рассчитываются как `0.70 × expert + 0.30 × data` с сохранением экспертного знака.

## Сценарии JSON

Встроены инерционный сценарий, ограниченный бюджет, ухудшение дорог, приоритет общественного транспорта, цифровая мобильность и повышение безопасности. Прямого ряда цифровизации нет, поэтому он явно обозначен как прокси регулярности, скорости и загруженности.

Пользовательский сценарий:

```json
{
  "version": 1,
  "id": "limited-budget-custom",
  "label": "Ограниченный бюджет",
  "description": "Пользовательская проверка",
  "mode": "adapted",
  "horizon": 8,
  "impulses": {
    "road_budget_execution": -0.15,
    "road_repair": -0.10
  }
}
```

Значения импульсов ограничены диапазоном `[-0.30, 0.30]`. Пользовательские сценарии сохраняются в PostgreSQL: основные поля — в отдельных колонках, `impulses` — в `JSONB`. Каждый сценарий имеет владельца; встроенные сценарии доступны всем ролям и не могут быть перезаписаны.

Пользователь или администратор может изменить импульсы, режим и горизонт прямо в интерфейсе и нажать «Сохранить». Для встроенного сценария создаётся пользовательская копия с новым идентификатором; для ранее сохранённого сценария обновляется существующая запись в PostgreSQL.

Основные API-маршруты после входа:

- `GET /api/metadata`, `/api/history`, `/api/indices`, `/api/evaluation`;
- `GET /api/fcm?mode=adapted`;
- `GET/POST/PUT/DELETE /api/scenarios`;
- `GET/PUT /api/scenarios/{id}/shares` — выбор наблюдателей для пользовательского JSON;
- `POST /api/simulate`.

## Авторизация и роли

Используется локальный вход по логину и паролю без OIDC. Пароли хранятся как Argon2id-хеши, серверные сессии — в PostgreSQL, изменяющие запросы защищены CSRF-токеном.

- `observer` / Наблюдатель — просматривает аналитику, встроенные сценарии и JSON, которыми с ним адресно поделились;
- `user` / Пользователь — создаёт, изменяет и удаляет свои JSON-сценарии, а также выбирает наблюдателей для просмотра;
- `admin` / Администратор — видит все сценарии вместе с именем и логином владельца, управляет пользователями и просматривает аудит.

Публичной регистрации нет. Начальные демонстрационные аккаунты создаются командой `python -m app.manage seed-demo-users`, пароли берутся из `.env`.

## Бизнес-интерпретация

После запуска сценария интерфейс показывает относительно инерционного прогноза:

- изменение безопасности, регулярности, доступности и итоговой мобильности в процентах;
- изменение тех же индексов в пунктах и значения «было → стало»;
- расчётное снижение количества ДТП;
- рейтинг трёх модельных бюджетных направлений по выбранной бизнес-цели.

Бюджетный рейтинг сравнивает одинаковое стандартизованное воздействие на исполнение дорожного, транспортного и безопасностного бюджета. Это приоритизация по поведению модели, а не финансовый ROI и не расчёт в рублях.

## Структура интерфейса

Аналитические разделы расположены в порядке, удобном для демонстрации заказчику:

1. **Лаборатория FCM-сценариев** — позволяет сравнить решения, увидеть эффект в процентах, сохранить JSON и адресно открыть его наблюдателям.
2. **Чувствительность** — ранжирует направления по ожидаемой силе влияния на выбранную цель.
3. **Текущее состояние** — даёт краткую сводку последнего доступного периода и исходную точку сравнения.
4. **История и сезонность** — помогает отличить устойчивую динамику от разового или сезонного изменения.
5. **Сводные индексы** — объединяет исходные показатели в понятные оценки от 0 до 100.
6. **Проверка моделей** — показывает качество прогноза на уже известных данных и границы неопределённости.
7. **Карта связей FCM** — наглядно объясняет связи между решениями, городскими факторами и целевыми показателями.

Все разделы 1–7 при открытии страницы свёрнуты. Под заголовком каждого находится заметная кнопка «Развернуть раздел» с поясняющим текстом; переход по ссылке из верхнего меню автоматически раскрывает нужный раздел. Лаборатория стоит первой, а чувствительность — непосредственно под ней. При раскрытии размеры графиков и FCM-карты пересчитываются под доступную ширину. Рядом с каждым заголовком есть подсказка «Что показывает?» — при наведении или фокусе она объясняет назначение блока нетехническим языком. Раздел 8 «Администрирование» показывается только администратору.

## Полная установка и запуск

### Что должно быть установлено

Для рекомендуемого запуска нужны:

- Windows 10/11 с WSL 2 и запущенным Docker Desktop;
- Git, если проект получается из репозитория;
- доступ к Docker Hub при первой сборке для загрузки базовых образов Python и PostgreSQL.

Python и виртуальное окружение для полного Docker-запуска не требуются: зависимости из `requirements.txt` устанавливаются внутрь образа приложения. `.venv` нужен только для локальной разработки по варианту 2.

Docker Desktop не устанавливается через `requirements.txt` и не входит в `.venv`. На Windows сначала установите и запустите [Docker Desktop](https://docs.docker.com/desktop/setup/install/windows-install/), затем проверьте:

```powershell
docker version
docker compose version
```

Если Docker Desktop завершает работу с сообщением `Access denied to Docker`, текущий пользователь не включён в локальную группу `docker-users`. Запустите PowerShell **от имени администратора**, выполните команду и затем выйдите из Windows и войдите снова:

```powershell
Add-LocalGroupMember -Group docker-users -Member $env:USERNAME
```

Членство в `docker-users` даёт возможность управлять контейнерами и должно предоставляться только доверенному локальному пользователю.

### Вариант 1. Весь проект в Docker Compose

Рекомендуемый способ для передачи и демонстрации. FastAPI и PostgreSQL запускаются в отдельных контейнерах, миграции и создание трёх аккаунтов выполняются автоматически.

Получите проект и перейдите в его каталог. Если репозиторий уже скачан, достаточно выполнить только `Set-Location`:

```powershell
git clone https://github.com/Cosmic-Man/neuro_simulator_smolensk.git
Set-Location neuro_simulator_smolensk
```

Создайте локальную конфигурацию, откройте её и замените пароль PostgreSQL и три демонстрационных пароля. Файл `.env` не попадёт в Git:

```powershell
Copy-Item .env.example .env
notepad .env
```

Запустите сборку и дождитесь состояния `healthy` у обоих сервисов:

```powershell
docker compose config --quiet
docker compose up --build -d
docker compose ps
```

Проверка API и последние сообщения приложения:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/health
docker compose logs --tail 100 app
```

Интерфейс: <http://127.0.0.1:8000>. Документация API: <http://127.0.0.1:8000/docs>.

Демонстрационные логины: `observer`, `user`, `admin`. Пароли заданы в локальном `.env`; перед передачей проекта их следует заменить.

Остановка без удаления данных:

```powershell
docker compose down
```

Первый запуск может занять несколько минут из-за загрузки и сборки образов. Последующие запуски используют локальный кэш Docker.

### Вариант 2. FastAPI из `.venv`, PostgreSQL в Docker

Удобен для разработки и отладки Python-кода:

```powershell
Copy-Item .env.example .env

```

Тот же процесс автоматизирован скриптами:

```powershell

```

Если в `runtime/scenarios` остались JSON-файлы старой версии, после миграции их можно однократно перенести владельцу:

```powershell
python -m app.manage import-scenarios --owner admin
```

### Что хранится в Git, а что остаётся только у клиента

В Git хранятся исходный код, миграции схемы, `compose.yaml`, `Dockerfile` и безопасный шаблон `.env.example`. В Git **не должны** попадать:

- рабочий `.env` с паролями;
- содержимое PostgreSQL;
- пользовательские JSON-сценарии, учётные записи, сессии и журнал аудита;
- локальные каталоги `postgres_data`, `pgdata`, `runtime`, `backups`;
- дампы `*.dump`, `*.backup`, `*.sql`, `*.sql.gz`, файлы SQLite и локальные файлы `pgpass`.

PostgreSQL хранит данные в именованном Docker volume `neuro-simulator-smolensk_postgres_data`, расположенном вне каталога репозитория. Поэтому `git add .`, `git pull`, пересборка приложения и `docker compose down` не отправляют и не удаляют клиентские данные. Правила продублированы в `.gitignore` и `.dockerignore`. Не используйте `git add -f` для `.env`, дампов или каталогов данных.

В репозитории находятся все файлы развёртывания:

- `Dockerfile` и `.dockerignore` — образ FastAPI без локальных секретов и `.venv`;
- `compose.yaml` — отдельные сервисы приложения и PostgreSQL, health-check и постоянный volume;
- `docker-entrypoint.sh` — автоматические миграции и начальные аккаунты;
- `.env.example` — безопасный шаблон локальных переменных;
- `alembic.ini` и `migrations/` — версионирование схемы PostgreSQL.

По умолчанию PostgreSQL доступен на `127.0.0.1:5432`. Если порт занят, измените `POSTGRES_PORT` в `.env`, например на `5433`.

### Обновление установленного проекта

Перед обновлением рекомендуется создать резервную копию. Затем получите изменения и пересоберите только изменившиеся слои. При старте приложение автоматически применит новые миграции PostgreSQL:

```powershell
git pull
docker compose up --build -d
docker compose ps
docker compose logs --tail 100 app
```

Обновление контейнера не меняет именованный volume PostgreSQL. Если менялись только значения `.env`, пересоздайте контейнеры без удаления данных:

```powershell
docker compose up -d --force-recreate
```

### Резервная копия PostgreSQL

Каталог `backups` исключён из Git. Следующие команды создают сжатый дамп внутри контейнера и копируют его на компьютер клиента без передачи в репозиторий:

```powershell
New-Item -ItemType Directory -Force .\backups | Out-Null
$backupName = "neuro_simulator_{0}.backup" -f (Get-Date -Format "yyyyMMdd_HHmmss")
docker compose exec postgres sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc -f /tmp/neuro_simulator.backup'
docker compose cp postgres:/tmp/neuro_simulator.backup ".\backups\$backupName"
docker compose exec postgres rm -f /tmp/neuro_simulator.backup
```

Храните резервные копии отдельно от каталога проекта. Восстановление дампа перезаписывает текущие данные и должно выполняться ответственным администратором после проверки выбранного файла.

### Полезные команды

```powershell
# Проверить состояние и посмотреть логи приложения
docker compose ps
docker compose logs -f app

# Пересобрать приложение после изменения исходников
docker compose up --build -d app

# Удалить контейнер, сохранив данные в Docker volume
docker compose down

# Удалить контейнер вместе с локальными данными PostgreSQL
# Внимание: эта команда необратимо удаляет базу демонстрационного проекта
docker compose down -v
```

`docker compose down -v` — единственная из приведённых штатных команд, которая удаляет именованный volume с клиентскими данными. Не используйте её при обновлении или обычной остановке.

Значения из `.env.example` предназначены только для локальной демонстрации. Для внешнего развёртывания пароли необходимо заменить, `SESSION_COOKIE_SECURE` включить при работе через HTTPS, а PostgreSQL не следует публиковать напрямую в интернет.

## Тесты

```powershell
.\.venv\Scripts\python.exe -m unittest discover -s tests -v
```

Тесты используют изолированную SQLite только как быстрый тестовый backend. Они проверяют Excel-схему, отсутствие временной утечки, нечёткие функции, FCM/ANFIS, роли, CSRF, владение сценариями, проценты и бюджетный рейтинг. Итоговая миграция дополнительно проверяется на реальном PostgreSQL через Compose.
