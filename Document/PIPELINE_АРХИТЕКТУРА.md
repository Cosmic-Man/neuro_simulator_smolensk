# Архитектура приложения на основе Pipeline.ipynb

## Статус

Канонический расчётный файл проекта — `colab/colab_kirill/Pipeline.ipynb`. Notebook версионируется в Git вместе с кодом, а его runtime-правила поставляются отдельными JSON-файлами. Если старые документы расходятся с этой схемой, приоритет имеет этот документ и код приложения.

## Поток данных

```text
smolensk_dataset_shared.xlsx: 80 кварталов × 31 показатель
  → проверка двухуровневой схемы и порядка столбцов
  → log1p для 6 показателей
  → IQR-проверка скорости городского транспорта
  → исключение выброса 2018Q3
  → 79 строк: train 51 / validation 16 / test 12
  → 8 нечётких свёрток Mamdani
  → линейный target по 8 экспертным весам
  → RobustScaler
  → Seasonal Naive / Ridge Regression / ANFIS
```

FCM не входит в расчётный notebook. В приложении она сохранена как отдельное демонстрационное расширение: получает показатели, сформированные Pipeline, и рассчитывает управленческие сценарии безопасности, регулярности и доступности.

## Нечёткие индексы

Pipeline рассчитывает восемь индексов:

1. качество современной городской среды;
2. качество дорог дорожного комплекса;
3. благополучие дорог дорожного комплекса;
4. удовлетворённость доступной средой;
5. качество общественных пространств;
6. качество дорог общественного транспорта;
7. благополучие дорог общественного транспорта;
8. качество парковок и безопасности движения.

Диапазоны и параметры треугольных функций перенесены в `app/fuzzy.py` в исходных единицах notebook. Mamdani использует минимум для активации, максимум для агрегации и centroid на сетке из 1000 точек.

Трапециевидная функция поддерживается вычислительным классом, но актуальные переменные Pipeline используют треугольники. В интерфейсе трапеция показана как понятное сравнение возможных форм экспертной оценки.

## Итоговый target

Исходные веса:

```text
0.6 / 0.6 / 0.7 / 0.2 / 0.5 / 0.7 / 0.9 / 0.4
```

Они делятся на сумму `4.6`. Итог — прямая взвешенная сумма восьми нечётких индексов без повторной min-max-нормализации.

## Сравнение моделей

- Seasonal Naive повторяет значение четырёхквартальной давности.
- Ridge использует лаги 1, 2 и 4 квартала и среднее за предыдущие четыре квартала.
- ANFIS получает восемь индексов после RobustScaler, использует шесть гауссовых правил и линейные консеквенты.

В NumPy-переносе ANFIS используются основные параметры notebook: `epochs=500`, `learning_rate=0.05`, `patience=70`, Adam и KMeans-инициализация центров. Линейные консеквенты инициализируются известной формулой target для воспроизводимости.

## Раздел 8 интерфейса

Раздел «Анализ данных Pipeline» доступен наблюдателю, пользователю и администратору. Он показывает:

- коробки с усами по тематическим группам;
- периоды и значения статистических выбросов;
- функции принадлежности любого входа или выхода восьми индексов;
- сравнение треугольной и трапециевидной формы;
- нормированные веса итоговой линейной свёртки.

Данные выдаёт защищённый маршрут `GET /api/analysis`.

## Раздел 9: редактор XLSX

В `datasets_ready` находятся основной файл на 80 кварталов и демонстрационная копия с дополнительным `2026Q1`, у которого 31 показатель равен нулю. Заказчик видит активный источник и последние строки. Пользователь и администратор могут выбрать файл, изменить полный набор показателей квартала или добавить следующий квартал; наблюдатель работает в режиме чтения.

Backend принимает только имена XLSX внутри `datasets_ready`, запрещает отрицательные и бесконечные значения, записывает временную копию и прогоняет её через весь Pipeline. Исходный файл заменяется атомарно только после успешной проверки. В Compose папка подключена как bind mount, поэтому изменения переживают пересборку контейнера.

## Пять рекомендаций заказчику

В лаборатории можно выбрать одну из четырёх бизнес-целей или один из восьми индексов Pipeline. Для выбранной цели всегда выводится ровно пять действий. Бизнес-цели ранжируются по чувствительности FCM; для индексов используются прямые входы их канонических JSON-правил и смежные факторы карты. Для индекса парковок выводятся исполнение бюджета, ремонт, нормативное состояние, срок устранения дефектов и связанная организация безопасных переходов.

## Сценарное расширение

Встроено 25 сценариев:

- 10 точечных улучшений `+1`;
- 3 прямых перераспределения `+1` одному и `−1` суммарно остальным;
- 2 обратных перераспределения `−1` одному и `+1` суммарно остальным;
- 8 сценариев заданных причинных связей;
- инерционный и пользовательский варианты.

Импульсы ограничены диапазоном `[-1, 1]`. Пользовательские JSON хранятся в PostgreSQL и не попадают в Git.

## Канонические JSON-правила

В Git включены все внешние JSON-файлы, которые загружает `Pipeline.ipynb`:

- `urban_city_area_rules.json`;
- `transport_r1_rules.json`;
- `transport_r2_rules.json`;
- `available_area_rules.json`;
- `public_spaces_rules.json`;
- `parking_and_security_rules.json`.

Backend загружает их непосредственно из `colab/colab_kirill`, проверяет структуру каждого правила и не использует старые встроенные таблицы. Файл `transport_r1_rules.json` содержит 235 правил и применяется к двум дорожным подсистемам; `transport_r2_rules.json` содержит 125 правил и также применяется дважды. Вместе восемь нечётких систем исполняют 1 176 правил. Контрольные первая и последняя строки всех восьми индексов, а также target `14.556916` и `66.714047`, совпадают с outputs notebook.
