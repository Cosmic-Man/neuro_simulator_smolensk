## 1. Назначение документа

Этот документ описывает следующий этап развития демонстрационного проекта:

- внедрение PostgreSQL;
- добавление локальной аутентификации и трёх ролей;
- перенос пользовательских JSON-сценариев FCM в базу данных;
- отображение результата сценария в понятных для бизнеса процентах;
- модельная оценка приоритетных направлений бюджета.

Проект остаётся демонстрационным аналитическим прототипом. На этом этапе не планируются корпоративный SSO, OIDC, Kubernetes, отказоустойчивый кластер и полноценная промышленная инфраструктура.

## 2. Принятые архитектурные решения

### 2.1. Формат запуска

Основной вариант запуска остаётся простым:

```text
браузер → FastAPI → PostgreSQL
              ↓
       расчётная модель FCM/ANFIS
```

FastAPI можно продолжать запускать через виртуальное окружение и `run_local.ps1`.

PostgreSQL предлагается поднимать одним из двух способов:

1. через Docker Compose — рекомендуемый воспроизводимый вариант для демонстрации;
2. через установленный локально PostgreSQL — запасной вариант без Docker.

Контейнеризация всего приложения не должна блокировать разработку. На первом этапе достаточно контейнера PostgreSQL с постоянным томом данных. После завершения PostgreSQL, ролей и бизнес-интерфейса проект целесообразно упаковать полностью, чтобы демонстрацию можно было запустить одной командой.

### 2.2. Два этапа работы с Docker

Docker Desktop и Docker Compose являются системными инструментами. Они не устанавливаются через `requirements.txt` и не входят в `.venv`. Перед запуском пользователь устанавливает Docker Desktop и проверяет:

```powershell
docker version
docker compose version
```

Предлагается два последовательных этапа.

**Этап разработки:**

```text
FastAPI из .venv → PostgreSQL в Docker
```

Такой режим удобен для отладки Python-кода: приложение запускается привычным `run_local.ps1`, а PostgreSQL воспроизводимо поднимается из `compose.yaml`.

**Этап передачи демонстрации:**

```text
Docker Compose
├─ app       отдельный контейнер FastAPI
└─ postgres  отдельный контейнер PostgreSQL
```

FastAPI и PostgreSQL не следует помещать в один контейнер. Один Compose-проект управляет двумя сервисами, их внутренней сетью, порядком запуска и постоянным томом базы данных.

Итоговая команда запуска демонстрации:

```powershell
Copy-Item .env.example .env
docker compose up --build
```

Полная упаковка не является перебором: она упрощает передачу проекта и исключает различия в версиях Python и системных библиотек. При этом её следует выполнять после стабилизации функциональности, чтобы не отлаживать одновременно бизнес-логику и контейнерное окружение.

### 2.3. Аутентификация

Для демонстрационной версии используется локальный вход по логину и паролю.

- Пароли хранятся только в виде Argon2id-хеша.
- После входа сервер создаёт непрозрачную сессию и сохраняет её в PostgreSQL.
- В браузер передаётся только cookie с идентификатором сессии.
- Cookie имеет флаги `HttpOnly` и `SameSite=Lax`.
- В локальном HTTP-режиме `Secure` отключён; при HTTPS-развёртывании он обязателен.
- Публичной регистрации нет: демонстрационные аккаунты создаются командой инициализации, остальные — администратором.
- Изменяющие запросы защищаются CSRF-токеном и проверкой источника запроса.

OIDC не входит в первую версию. Код проверки пользователя следует отделить от бизнес-логики, чтобы внешний провайдер входа можно было подключить позднее без изменения модели ролей и таблицы сценариев.

## 3. PostgreSQL

### 3.1. Технологии

Планируемый стек:

- PostgreSQL;
- SQLAlchemy 2;
- `psycopg`;
- Alembic для миграций;
- `pwdlib[argon2]` для паролей.

Строка подключения задаётся переменной окружения:

```text
DATABASE_URL=postgresql+psycopg://user:password@localhost:5432/neuro_simulator
```

Секреты и пароли базы данных не добавляются в Git.

### 3.2. Таблица пользователей

```text
users
├─ id                 UUID, первичный ключ
├─ username           уникальный нормализованный логин
├─ display_name       отображаемое имя
├─ password_hash      Argon2id-хеш
├─ role               observer | user | admin
├─ is_active          разрешён ли вход
├─ created_at
├─ updated_at
└─ last_login_at
```

Роль хранится как строка с ограничением допустимых значений. Это роли приложения, а не отдельные пользователи или роли PostgreSQL.

### 3.3. Таблица сессий

```text
sessions
├─ id                 UUID
├─ token_hash         уникальный хеш случайного токена
├─ user_id            ссылка на users
├─ csrf_token_hash
├─ created_at
├─ expires_at
└─ revoked_at
```

Исходный токен сессии в базе не хранится. Выход из системы устанавливает `revoked_at`, после чего cookie больше не даёт доступ.

### 3.4. Таблица сценариев FCM

```text
scenarios
├─ id                 UUID, внутренний идентификатор
├─ owner_id           ссылка на users
├─ slug               текущий JSON-параметр id
├─ label
├─ description
├─ mode               expert | adapted
├─ horizon            1..20 кварталов
├─ impulses           JSONB
├─ schema_version     версия JSON-формата
├─ created_at
├─ updated_at
└─ UNIQUE(owner_id, slug)
```

Пользовательские JSON-сценарии FCM становятся постоянными данными PostgreSQL. API продолжает принимать и возвращать привычный формат:

```json
{
  "version": 1,
  "id": "balanced-mobility-2030",
  "label": "Сбалансированная мобильность — 2030",
  "description": "Описание сценария",
  "mode": "adapted",
  "horizon": 12,
  "impulses": {
    "road_budget_execution": 0.12,
    "transit_budget_execution": 0.14
  }
}
```

При записи API раскладывает основные поля по колонкам, а словарь `impulses` сохраняет как `JSONB`.

Встроенные сценарии остаются в коде:

- они доступны всем ролям;
- их нельзя изменить или удалить через API;
- они версионируются вместе с приложением.

Экспертная и адаптированная матрицы FCM не становятся постоянным источником данных в PostgreSQL. Они рассчитываются из кода и датасета при запуске. При необходимости воспроизводимости в будущем можно сохранять снимок матрицы вместе с конкретным результатом симуляции.

### 3.5. Адресный доступ наблюдателей

```text
scenario_shares
├─ id                 UUID
├─ scenario_id        ссылка на scenarios
├─ observer_id        ссылка на users с ролью observer
├─ created_at
└─ UNIQUE(scenario_id, observer_id)
```

Владелец выбирает одного или нескольких активных наблюдателей. Наблюдатель получает право просмотра, экспорта и запуска симуляции, но не может изменить или удалить JSON. Удаление сценария автоматически удаляет все выданные доступы.

### 3.6. Журнал действий

```text
audit_log
├─ id
├─ user_id
├─ action
├─ entity_type
├─ entity_id
├─ details            JSONB без паролей и токенов
└─ created_at
```

Для демонстрации достаточно фиксировать:

- успешный и неуспешный вход;
- выход;
- создание, изменение и удаление сценария;
- изменение роли или статуса пользователя.

### 3.7. Перенос существующих сценариев

В `runtime/scenarios` уже находятся пользовательские JSON-файлы. Нужна отдельная идемпотентная команда импорта:

```text
python -m app.manage import-scenarios --owner <admin_username>
```

Правила импорта:

- сценарии проверяются существующим валидатором;
- владельцем становится указанный администратор;
- повторный запуск не создаёт дубликаты;
- конфликтующий `slug` пропускается с понятным сообщением;
- исходные JSON-файлы не удаляются и остаются резервной копией.

После успешного внедрения файловое хранилище перестаёт использоваться приложением для новых сценариев.

## 4. Три степени доступа

### 4.1. Наблюдатель — `observer`

Наблюдатель может:

- просматривать исторические данные, индексы и метрики моделей;
- просматривать когнитивную карту;
- использовать встроенные сценарии;
- просматривать пользовательские JSON, которыми владелец поделился именно с ним;
- менять параметры расчёта в интерфейсе;
- запускать симуляцию без сохранения результата;
- видеть бизнес-проценты и бюджетные приоритеты.

Наблюдатель не может:

- сохранять пользовательские сценарии;
- изменять или удалять сценарии;
- видеть остальные сценарии пользователей, к которым ему не выдан доступ;
- управлять аккаунтами и ролями.

### 4.2. Пользователь — `user`

Пользователь получает все возможности наблюдателя и дополнительно может:

- создавать собственные сценарии;
- загружать JSON-сценарии;
- просматривать только свои и встроенные сценарии;
- изменять и удалять собственные сценарии;
- выгружать собственный сценарий в совместимом JSON-формате.
- выбирать активных наблюдателей, которым разрешён просмотр конкретного JSON.

Пользователь не может изменять чужие и встроенные сценарии.

### 4.3. Администратор — `admin`

Администратор получает полный доступ и может:

- создавать пользователей;
- назначать роли `observer`, `user`, `admin`;
- блокировать и разблокировать аккаунты;
- сбрасывать пароль пользователя;
- просматривать и удалять любой пользовательский сценарий;
- видеть у каждого JSON имя и логин владельца;
- просматривать журнал действий;
- выполнять импорт старых JSON-сценариев.

Удалять собственную последнюю активную учётную запись администратора нельзя.

### 4.4. Матрица доступа

| Операция | Наблюдатель | Пользователь | Администратор |
|---|:---:|:---:|:---:|
| Просмотр аналитики | Да | Да | Да |
| Запуск симуляции | Да | Да | Да |
| Просмотр встроенных сценариев | Да | Да | Да |
| Просмотр пользовательского JSON | Только выданные | Свои | Все, с владельцем |
| Сохранение сценария | Нет | Свой | Любой |
| Изменение и удаление сценария | Нет | Свой | Любой |
| Выдача доступа наблюдателю | Нет | Для своего | Для любого |
| Управление пользователями | Нет | Нет | Да |
| Просмотр аудита | Нет | Нет | Да |

Ограничения проверяются в FastAPI. Скрытие кнопки в браузере не считается защитой.

## 5. Изменения API

### 5.1. Аутентификация

```text
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
POST /api/auth/change-password
```

`GET /api/auth/me` возвращает идентификатор, имя, роль и CSRF-токен текущей сессии. Пароль, его хеш и токен сессии никогда не возвращаются.

### 5.2. Администрирование

```text
GET   /api/admin/users
POST  /api/admin/users
PATCH /api/admin/users/{user_id}
POST  /api/admin/users/{user_id}/reset-password
GET   /api/admin/audit
```

Эти маршруты доступны только роли `admin`.

### 5.3. Сценарии

```text
GET    /api/scenarios
POST   /api/scenarios
PUT    /api/scenarios/{scenario_id}
DELETE /api/scenarios/{scenario_id}
GET    /api/scenarios/{scenario_id}/export
GET    /api/scenarios/{scenario_id}/shares
PUT    /api/scenarios/{scenario_id}/shares
```

`GET /api/scenarios` возвращает:

- встроенные сценарии;
- собственные сценарии пользователя;
- для наблюдателя — только адресно предоставленные JSON с владельцем;
- для администратора — все пользовательские сценарии с владельцами.

Попытка обратиться к чужому сценарию возвращает `404`, чтобы не раскрывать его существование. Недостаточная роль для общей операции возвращает `403`, отсутствие сессии — `401`.

## 6. Бизнес-представление результата

### 6.1. Задача

Сейчас симулятор показывает линии и изменение в пунктах. Для демонстрации бизнес-пользователю нужны крупные ответы на вопросы:

- насколько повысится безопасность движения;
- насколько повысится регулярность общественного транспорта;
- насколько повысится транспортная доступность.

### 6.2. Формат расчёта

Для каждой метрики сравниваются конечные точки сценария и инерционного прогноза на одинаковом горизонте.

```text
абсолютное изменение = сценарий − инерционный прогноз

относительное изменение, % =
    абсолютное изменение / |инерционный прогноз| × 100
```

Показываются оба значения, потому что процент и процентный пункт — разные величины.

Пример отображения:

```text
Регулярность транспорта
+4,8%
+3,1 п.п. · 64,5% → 67,6%
```

Для индексов безопасности и доступности подпись использует индексные пункты:

```text
Безопасность движения
+6,2%
+4,0 пункта · 64,3 → 68,3
```

Если значение инерционного прогноза практически равно нулю, относительный процент не рассчитывается. API возвращает `null`, а интерфейс показывает `н/д` и абсолютное изменение.

Для безопасности дополнительно выводится понятный негативный показатель:

```text
Расчётное снижение ДТП: 5,4%
```

Здесь улучшение рассчитывается в обратном направлении:

```text
снижение ДТП, % =
    (ДТП baseline − ДТП scenario) / |ДТП baseline| × 100
```

### 6.3. Новые поля ответа симуляции

`POST /api/simulate` сохраняет существующие поля и дополнительно возвращает:

```json
{
  "summary": {
    "safety": {
      "baseline": 64.3,
      "scenario": 68.3,
      "delta_points": 4.0,
      "relative_change_percent": 6.22
    },
    "regularity": {},
    "accessibility": {},
    "integrated_mobility": {},
    "accidents": {
      "baseline": 8.1,
      "scenario": 7.7,
      "delta": -0.4,
      "improvement_percent": 4.94
    }
  }
}
```

Значения в примере иллюстративные и не должны фиксироваться в коде или документации как реальный прогноз.

### 6.4. Изменения интерфейса

Над четырьмя существующими графиками появляется блок «Ожидаемый эффект»:

- карточка «Безопасность движения»;
- карточка «Регулярность транспорта»;
- карточка «Транспортная доступность»;
- компактная карточка «Итоговый индекс мобильности».

Правила оформления:

- крупное значение — относительное изменение в процентах;
- ниже — изменение в пунктах и значения `было → стало`;
- положительный эффект — зелёный цвет и стрелка вверх;
- отрицательный эффект — красный цвет и стрелка вниз;
- изменение меньше установленного порога округления — нейтральное состояние;
- на карточке всегда указывается горизонт прогноза;
- рядом размещается подпись «Оценка модели относительно инерционного сценария».

Графики не удаляются: карточки дают быстрый вывод, а графики показывают динамику и не позволяют скрыть промежуточные изменения.

## 7. Модельная оценка бюджетных приоритетов

### 7.1. Что уже есть в модели

В FCM присутствуют три бюджетных управляющих фактора:

1. исполнение дорожного бюджета;
2. исполнение бюджета общественного транспорта;
3. исполнение бюджета безопасности.

Датасет содержит проценты исполнения программ, но не содержит:

- суммы финансирования в рублях;
- стоимости отдельных мероприятий;
- минимальных обязательных расходов;
- фактического результата на один вложенный рубль.

Поэтому первая версия не может честно рассчитать финансовый ROI или оптимальное распределение конкретной суммы денег.

### 7.2. Что можно показать корректно

В интерфейс добавляется блок «Куда направить ресурсы».

Для каждого из трёх бюджетных узлов выполняется отдельный прогноз:

```text
baseline: без внешнего воздействия
вариант: постоянный модельный импульс +0,07
горизонт: выбранное пользователем число кварталов
```

`+0,07` — стандартизованное безразмерное воздействие FCM. Его нельзя называть 7%, семью процентными пунктами или денежной суммой.

Для каждого направления выводится матрица эффекта:

| Направление | Безопасность | Регулярность | Доступность | Общий индекс |
|---|---:|---:|---:|---:|
| Дорожная программа | модельный результат | модельный результат | модельный результат | модельный результат |
| Общественный транспорт | модельный результат | модельный результат | модельный результат | модельный результат |
| Безопасность движения | модельный результат | модельный результат | модельный результат | модельный результат |

В каждой ячейке показываются относительный процент и изменение в пунктах.

### 7.3. Выбор бизнес-цели

Пользователь выбирает цель ранжирования:

- сбалансированный результат — по итоговому индексу мобильности;
- максимизация безопасности;
- максимизация регулярности;
- максимизация доступности.

Направления сортируются по изменению выбранной метрики. Первое направление выделяется как «Модельный приоритет».

Формулировка рекомендации:

```text
При одинаковом модельном воздействии на выбранном горизонте
наибольший прирост показателя «Доступность» даёт направление
«Общественный транспорт».
```

Нельзя использовать формулировки:

- «гарантированная экономия»;
- «окупаемость бюджета»;
- «точный финансовый эффект»;
- «необходимо вложить конкретную сумму».

Под таблицей обязательно показывается предупреждение:

> Рейтинг отражает поведение модели при одинаковом стандартизованном воздействии. Он помогает сравнивать направления, но не заменяет финансово-экономическое обоснование и не является расчётом эффективности в рублях.

### 7.4. Полный бюджетный оптимизатор — будущий этап

Для настоящего перераспределения фиксированной суммы потребуются дополнительные входные данные:

- доступный бюджет;
- текущие и минимальные расходы по направлениям;
- стоимость единицы ремонта, перехода и другого мероприятия;
- ограничения по срокам и мощности выполнения;
- историческая связь денежных затрат с физическим результатом.

После появления этих данных можно будет рассчитывать:

```text
эффективность = прирост целевого индекса / 1 млн рублей
```

и искать распределение, которое максимизирует выбранный индекс при заданных ограничениях. Это отдельная задача оптимизации и не входит в текущий демонстрационный этап.

## 8. Изменения общего интерфейса

### 8.1. Вход и текущая роль

- До входа отображается отдельная компактная форма логина.
- В шапке показываются имя пользователя, роль и кнопка выхода.
- При истечении сессии приложение возвращает пользователя на форму входа.
- Для наблюдателя скрываются загрузка и сохранение JSON.
- Для пользователя отображаются личные сценарии и выбор наблюдателей.
- Для администратора явно показывается владелец JSON, появляется раздел «Пользователи» и просмотр аудита.

### 8.2. Общая структура страницы

Порядок аналитических разделов:

1. «Лаборатория FCM-сценариев» — расчёт, проценты, бюджетные приоритеты, сохранение и адресный показ JSON;
2. «Чувствительность» — рейтинг факторов по силе ожидаемого влияния;
3. «Текущее состояние» — исходная управленческая сводка;
4. «История и сезонность» — динамика и устойчивость изменений;
5. «Сводные индексы» — агрегированные оценки от 0 до 100;
6. «Проверка моделей» — качество прогнозирования на известных данных;
7. «Карта связей FCM» — объяснение связей между управленческими решениями, факторами и целевыми показателями.

Разделы 1–7 изначально свёрнуты. Для каждого показывается отдельная кнопка «Развернуть раздел» и текст, прямо указывающий, что содержимое доступно по нажатию. Переход по якорной ссылке автоматически раскрывает целевой раздел. После раскрытия приложение повторно рассчитывает ширину графиков и FCM-карты, чтобы содержимое не выходило за границы гармошки.

У каждого заголовка 1–7 находится подсказка «Что показывает?». При наведении мыши или клавиатурном фокусе она объясняет пользу раздела языком заказчика, без описания программной реализации. Административный раздел сохраняет номер 8 и доступен только роли `admin`.

### 8.3. Приоритет лаборатории сценариев

Порядок информации в лаборатории сценариев:

1. настройки сценария;
2. бизнес-карточки ожидаемого эффекта;
3. бюджетные приоритеты;
4. графики динамики;
5. текстовое объяснение и применённые воздействия.

Это позволяет сначала увидеть управленческий вывод, а затем проверить его по графикам и объяснению модели.

## 9. Этапы внедрения

### Этап 1. Основа PostgreSQL

- добавить зависимости;
- вынести `DATABASE_URL` в конфигурацию;
- создать SQLAlchemy-модели;
- настроить Alembic;
- добавить опциональный Docker Compose только для PostgreSQL;
- добавить проверку соединения в health-check.

### Этап 2. Пользователи и роли

- реализовать создание начальных демонстрационных аккаунтов;
- добавить вход, выход и смену пароля;
- реализовать серверные сессии;
- добавить зависимости `current_user` и `require_role`;
- защитить маршруты API;
- добавить интерфейс входа и отображение роли.

### Этап 3. Сценарии в PostgreSQL

- заменить файловый `ScenarioStore` реализацией на PostgreSQL;
- сохранить текущую валидацию JSON;
- добавить владельца и проверки доступа;
- добавить адресный доступ к JSON для выбранных наблюдателей;
- реализовать изменение, удаление и экспорт;
- импортировать существующие JSON-файлы.

### Этап 4. Бизнес-проценты

- добавить серверный расчёт абсолютного и относительного эффекта;
- расширить контракт `/api/simulate`;
- добавить карточки безопасности, регулярности и доступности;
- добавить вторичный показатель снижения ДТП;
- сохранить существующие графики.

### Этап 5. Бюджетный рейтинг

- выполнить три изолированных расчёта с одинаковым импульсом;
- возвращать эффект по четырём целевым показателям;
- добавить выбор бизнес-цели;
- построить рейтинг и сравнительную таблицу;
- добавить методологическое предупреждение.

### Этап 6. Полная упаковка проекта в Docker

- добавить `Dockerfile` для FastAPI на основе компактного официального Python-образа;
- установить Python-зависимости внутри образа из `requirements.txt`;
- запускать приложение в контейнере не от пользователя `root`;
- добавить `.dockerignore` для `.git`, `.venv`, `.env`, runtime-файлов и Python-кэшей;
- расширить `compose.yaml` сервисом `app` и сохранить PostgreSQL отдельным сервисом;
- подключить `app` к PostgreSQL по внутреннему имени сервиса, а не через `localhost`;
- запускать миграции Alembic перед стартом Uvicorn;
- запускать Uvicorn внутри контейнера на `0.0.0.0:8000`, публикуя порт только на локальную машину;
- дождаться успешного health-check PostgreSQL перед запуском приложения;
- хранить данные PostgreSQL в именованном Docker volume;
- не встраивать `.env` и пароли в Docker-образ;
- обеспечить запуск всей демонстрации командами `docker compose up --build` и `docker compose down`.

### Этап 7. Проверка демонстрации

- подготовить сценарий показа для каждой роли;
- проверить гибридный запуск: PostgreSQL в Docker, FastAPI из `.venv`;
- проверить полный запуск двух сервисов через Docker Compose;
- убедиться, что перезапуск не удаляет пользователей и сценарии;
- проверить читаемость бизнес-карточек на небольшом экране;
- зафиксировать ограничения модели в интерфейсе.

## 10. Тестирование

### 10.1. PostgreSQL и миграции

- чистая база обновляется до последней миграции;
- повторное применение миграций безопасно;
- ограничения ролей, владельцев и уникальности работают;
- импорт JSON не создаёт дубликаты;
- данные сохраняются после перезапуска.

### 10.2. Аутентификация и роли

- правильный пароль создаёт сессию;
- неправильный пароль не раскрывает наличие пользователя;
- выход и истечение срока отзывают доступ;
- запрос без входа получает `401`;
- недостаточная роль получает `403`;
- пользователь не читает и не изменяет чужой сценарий;
- наблюдатель видит только выданные ему JSON и не может их сохранять или изменять;
- администратор видит владельца каждого пользовательского JSON;
- администратор управляет пользователями;
- нельзя отключить последнего активного администратора.

### 10.3. Бизнес-показатели

- правильно рассчитываются положительные и отрицательные изменения;
- изменение в пунктах не смешивается с относительным процентом;
- нулевой baseline возвращает `null` для относительного процента;
- снижение ДТП отображается как улучшение;
- округление API и интерфейса согласовано;
- горизонт на карточках совпадает с выбранным горизонтом.

### 10.4. Бюджетный рейтинг

- сравниваются только три бюджетных узла;
- для всех направлений используется одинаковый импульс и горизонт;
- смена бизнес-цели меняет сортировку, но не исходные результаты;
- первое направление соответствует максимальному эффекту выбранной метрики;
- экспертный и адаптированный режимы не смешиваются;
- предупреждение о модельной природе рейтинга всегда видно.

### 10.5. Регрессия

Все существующие тесты данных, нечёткой логики, FCM, ANFIS и сценарных направлений должны продолжить выполняться. Тесты файлового хранения заменяются тестами PostgreSQL и импорта JSON.

### 10.6. Docker

- `docker compose config` успешно проверяет итоговый Compose-файл;
- образы собираются из чистого Docker-кэша;
- PostgreSQL становится healthy до запуска FastAPI;
- миграции применяются к пустой базе автоматически;
- интерфейс открывается на `http://127.0.0.1:8000`;
- повторный запуск использует сохранённые данные из volume;
- удаление контейнеров без флага `-v` не удаляет базу;
- в собранный образ не попадают `.env`, `.venv` и локальные runtime-файлы.

## 11. Критерии готовности

Улучшение считается готовым, если:

- PostgreSQL запускается воспроизводимо и мигрируется одной командой;
- существуют демонстрационные аккаунты трёх ролей;
- сервер, а не только интерфейс, применяет ограничения доступа;
- пользовательские JSON-сценарии сохраняются в PostgreSQL и имеют владельца;
- существующие JSON можно импортировать и экспортировать;
- после симуляции показаны проценты безопасности, регулярности и доступности;
- рядом с процентом показано изменение в пунктах и значения `было → стало`;
- бюджетный блок ранжирует три направления по выбранной цели;
- интерфейс не выдаёт модельный рейтинг за финансовый ROI;
- проект запускается как из `.venv`, так и полностью через Docker Compose;
- FastAPI и PostgreSQL работают в отдельных контейнерах;
- все математические и новые интеграционные тесты проходят.

## 12. Что сознательно не входит в этот этап

- OIDC и корпоративный SSO;
- Kubernetes и облачная инфраструктура;
- публикация Docker-образов в реестр и автоматическое промышленное развёртывание;
- горизонтальное масштабирование;
- регистрация пользователей через интернет;
- восстановление пароля по электронной почте;
- расчёт денежного ROI без исходных финансовых данных;
- автоматическое принятие бюджетных решений без участия эксперта.

Такой объём сохраняет проект демонстрационным, но показывает полноценную архитектурную эволюцию: постоянное хранение данных, разграничение прав и понятную бизнес-интерпретацию результатов модели.
